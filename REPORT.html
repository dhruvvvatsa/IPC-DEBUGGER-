1. Project Overview

IPC Debugger is a modern, full-stack web application designed to help systems-programming students visualize, debug, and simulate Inter-Process Communication (IPC) mechanisms — namely Pipes, Message Queues, and Shared Memory. Traditional learning of IPC is heavily textbook-driven, and observing real-time behaviour of concurrent processes is difficult on a typical operating-system shell. This project bridges that gap by offering an interactive, browser-based dashboard where students can load pre-built scenarios (Producer-Consumer, Deadlock, Race Condition, Bottleneck, Pipeline) or build their own configurations from scratch.

The application includes a real-time graph visualization of processes and their IPC channels, animated data-flow packets, an event timeline, deadlock detection using wait-for-graph cycle analysis, performance charts (latency, throughput, queue depth), shared-memory race-condition highlighting, and AI-style fix suggestions. The entire tool runs in the browser with no kernel-level dependencies, making it portable, safe, and ideal for academic use.

3. Functionalities

3.1 Real-Time IPC Monitoring Dashboard

  • Processes rendered as draggable circular nodes; channels as curved, directional connections.
  • Animated coloured packets travel along channels to depict live message flow.
  • Channels are colour-coded — Green = Normal, Yellow = Slow, Red = Blocked.
  • Live metrics (throughput, latency) printed on each connection.

3.2 Timeline View

  • Gantt-style execution lanes per process showing event markers over time.
  • Detailed event-log table with timestamps, type icons, source/target processes, and channel names.
  • Auto-scrolling to the latest event.

3.3 Deadlock Detection

  • Builds a Wait-For Graph from blocked processes and their blocked channels.
  • Runs a Depth-First Search to identify circular dependencies (cycles).
  • Highlights cycle members on the graph (pulsing red ring) and prints an explanation + fix suggestion.

3.4 Bottleneck & Performance Analysis

  • Sparkline charts for Message Rate and Average Latency (last 60 samples).
  • Per-channel queue-size charts.
  • Tabular channel-statistics view with status badges.
  • Bottleneck alerts ("slow consumer", "overloaded") with concrete suggestions.

3.5 Shared Memory Visualization

  • Memory segment displayed as a hex-grid of cells.
  • Last writer indicated by a coloured dot per cell.
  • Contested cells (race condition) blink red.
  • Suggestions to add pthread_mutex_lock() / unlock().

3.6 Interactive Simulation Mode

  • Manual creation of processes & channels via forms.
  • Choice of channel type — Pipe, Message Queue, Shared Memory.
  • Controls: Start, Pause, Step, Reset.
  • Speed multipliers: 0.5×, 1×, 2×, 4×.
  • Five pre-built demo scenarios.
  • Export full simulation state (processes, channels, events, metrics, issues) to JSON.

3.7 AI-Style Suggestions (Bonus)

  • Heuristic engine raises issues for high latency, queue overflow, deadlock, and race conditions.
  • Each issue carries a remediation tip — e.g., "Add backpressure", "Use try-lock instead of blocking lock".

7. Conclusion and Future Scope

The IPC Debugger successfully delivers an intuitive, visual, and interactive learning aid for Inter-Process Communication concepts. By abstracting away kernel complexity and focusing on visualization, the tool makes traditionally opaque OS behaviour — deadlocks, race conditions, slow consumers — visible and explainable. The pre-built scenarios serve as ready-to-use teaching aids, while the custom-build mode encourages experimentation. Performance metrics and AI-style suggestions further reinforce best practices in concurrent programming.

Future Scope

  • Real OS Hooks: Optionally connect to a Node.js backend that uses strace / ltrace to capture real syscall traces.
  • Recording & Replay: Persist simulations to SQLite and replay them frame-by-frame.
  • Multiplayer Mode: WebSocket-driven collaborative debugging where multiple students manipulate the same simulation.
  • Code Generation: Auto-generate equivalent C / Python source code from a visual configuration.
  • LLM Integration: Replace heuristic suggestions with a real Large Language Model for richer explanations.
  • Additional Mechanisms: Sockets, signals, semaphores, and condition variables.
  • Mobile Support: Touch-friendly graph manipulation.

A. AI-Generated Project Elaboration / Breakdown Report

The project was scoped and elaborated using an AI-assisted development workflow. Below is a condensed breakdown produced during the planning phase:

Goal: Build a full-stack web application that lets a developer or student visualize, debug, and simulate the three primary Inter-Process Communication mechanisms (Pipes, Message Queues, Shared Memory) without requiring real kernel-level instrumentation.

Architecture Decision: The originally suggested stack of Node.js + Express + WebSockets + SQLite was simplified to a pure client-side implementation. This is justified because (a) the simulation is deterministic and self-contained, (b) it removes any deployment friction for a university submission, and (c) the vite-plugin-singlefile plugin allows the entire app to be shipped as a single index.html file — perfect for grade-evaluation drop-boxes.

Module Breakdown (AI-derived):

  1. Type System — Centralised TypeScript interfaces guarantee consistent shapes across components.
  2. Simulation Engine — Pure-function tick() reducer in Zustand. Each tick mutates processes/channels/metrics and emits events. Speed is controlled by adjusting the JavaScript setInterval period.
  3. Visualization Layer — Native SVG (chosen over D3.js) keeps the bundle lean and the rendering predictable. Zoom/pan implemented via SVG viewBox, dragging via mouse-coordinate transforms.
  4. Detection Algorithms — Wait-For Graph cycle detection (Coffman 1971) implemented as DFS with an "in-stack" set. Race conditions detected via the heuristic of "two writers + one reader on the same shared segment within the same tick".
  5. UI Composition — Split-pane layout (Header / Graph / Sidebar / Timeline) chosen for familiarity (mirrors VS Code, Chrome DevTools).
  6. Heuristic AI Engine — Rules-based: latency > 100 ms ⇒ warning, queue ratio > 85% ⇒ error, cycle in WFG ⇒ critical. Each rule maps to a fixed suggestion string. Easily replaceable with an LLM call in the future.

Trade-offs Considered:

  • SVG vs Canvas — SVG won for declarative ease and accessibility (text remains selectable).
  • Redux vs Zustand — Zustand chosen for minimal boilerplate.
  • Real backend vs simulated — Simulated chosen to keep scope appropriate for a university project.

B. Problem Statement

Inter-Process Communication is a foundational topic in operating-systems courses, yet it remains one of the hardest for students to internalise. Traditional teaching relies on textbook diagrams and small C programs that print to the terminal — neither approach makes the dynamic behaviour of concurrent processes (message flow, queue build-up, deadlock formation, race conditions) visible. Existing professional debuggers (e.g., strace, ltrace, SystemTap) require root privileges and have steep learning curves, making them unsuitable for classroom use.

Objective: Design and implement a web-based IPC Debugger that:

  1. Visually represents processes and IPC channels as an interactive graph.
  2. Animates real-time data flow with colour-coded health indicators.
  3. Detects deadlocks automatically and explains them in plain English.
  4. Identifies bottlenecks and race conditions with actionable suggestions.
  5. Provides a sandbox where students can build, run, and step through their own scenarios.
  6. Works entirely in a browser with no kernel-level dependencies.

C.2   Deadlock Detection (DFS on Wait-For Graph)

// Inside src/store/useStore.ts — detectDeadlocks()
const waitFor: Record<string, string[]> = {};
blockedProcesses.forEach((p) => {
  waitFor[p.id] = [];
  channels.forEach((c) => {
    if (c.sourceId === p.id && c.status === 'blocked') {
      waitFor[p.id].push(c.targetId);
    }
  });
});

const visited = new Set<string>();
const inStack = new Set<string>();
const cycles: string[][] = [];

function dfs(node: string, path: string[]): void {
  visited.add(node);
  inStack.add(node);
  path.push(node);
  for (const neighbor of (waitFor[node] || [])) {
    if (!visited.has(neighbor))      dfs(neighbor, [...path]);
    else if (inStack.has(neighbor))  cycles.push(path.slice(path.indexOf(neighbor)));
  }
  inStack.delete(node);
}

tick: () => {
  const state = get();
  const tick  = state.simTick + 1;

  // 1. Update each process (CPU, memory, state machine)
  const updatedProcesses = state.processes.map((p) => ({
    ...p,
    cpu:    Math.max(0, Math.min(100, p.cpu + (Math.random() - 0.5) * 15)),
    state:  nextProcessState(p, scenario, tick),

  // 2. Update each channel (throughput, latency, queue, status)

}
